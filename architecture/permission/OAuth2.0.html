# OAuth 2.0

# 1. 介绍

OAuth 2.0 是一个行业的标准授权协议，用来授权第三方应用。

OAuth 2.0 的最终目的是为第三方应用颁发一个有时效性的令牌 token，使得第三方应用能够通过该令牌获取相关的资源。常见的场景就是：微信第三方授权登录。


# 2. 角色

OAuth 2.0 协议中有以下角色：

  • 资源所有者：允许访问受保护资源的实体。

  • 资源服务器：托管受保护资源的服务器。

  • 客户端：使用资源所有者的授权，代表资源所有者发起对受保护资源请求的第三方应用程序。如：web 网站，移动应用等。

  • 授权服务器：向客户端颁发令牌的服务器。

  • 用户代理：帮助资源所有者与客户端沟通的工具，一般为 web 浏览器，移动 APP 等。


# 3. 注册授权

客户端想要获得授权，需要先到服务商那注册，一般需要提供以下信息：

  • 应用名称

  • 应用网站

  • 用户允许或拒绝授权，应用程序重定向的地址 URL


注册成功之后，会从服务商那获取到自己应用的相关信息：

  • client_id：客户端的标识，是公开的。

  • client_secret：：客户端的秘钥，用来验证客户端身份，是保密的。


# 4. 授权类型

OAuth 2.0 一共有 4 种授权类型，分别用于不同的场景：

# 1）授权码

授权码（authorization code）方式，是第三方应用先申请一个授权码，然后再用该授权码获取令牌。


这种方式是最常用的流程，安全性也最高，它适用于那些有后端的应用。

授权码通过前端传送，令牌则是储存在后端，而且所有与资源服务器的通信都在后端完成。这样的方式可以避免令牌泄漏。

流程示例：

  1. A 网站提供一个授权链接 B，用户点击后就会跳转到 B 网站，授权用户数据给 A 网站使用。

    下面是 A 网站跳转 B 网站的一个示意链接：

    https://b.com/oauth/authorize?
  response_type=code&
  client_id=CLIENT_ID&
  redirect_uri=CALLBACK_URL&
  scope=read

    • response_type：值为 code，表示要求返回授权码。

    • client_id：客户端标识码，让 B 知道是谁在请求。

    • redirect_uri：是 B 接受或拒绝请求后的跳转网址。

    • scope：表示要求的授权范围（这里是只读）。


  1. 用户跳转后，B 网站会要求用户登录，然后询问是否同意给予 A 网站授权。

    用户表示同意，这时 B 网站就会跳回 redirect_uri 参数指定的网址。跳转时，会传回一个授权码：

    https://a.com/callback?code=AUTHORIZATION_CODE

  2. A 网站拿到授权码以后，就可以在后端 向 B 网站请求令牌。

    https://b.com/oauth/token?
 client_id=CLIENT_ID&
 client_secret=CLIENT_SECRET&
 grant_type=authorization_code&
 code=AUTHORIZATION_CODE&
 redirect_uri=CALLBACK_URL

  • client_id：客户端标识码，让 B 知道是谁在请求。

  • client_secret：客户端秘钥。

  • grant_type：值为 AUTHORIZATION_CODE，表示采用的授权方式是授权码。

  • code：上一步拿到的授权码。

  • redirect_uri：令牌颁发后的回调网址。


  1. B 网站收到请求以后，就会颁发令牌。

    具体做法是向 redirect_uri 指定的网址，发送一段 JSON 数据。

    {    
  "access_token":"ACCESS_TOKEN",
  "token_type":"bearer",
  "expires_in":2592000,
  "refresh_token":"REFRESH_TOKEN",
  "scope":"read",
  "uid":100101,
  "info":{...}
}

    上面 JSON 数据中，access_token 字段就是令牌，A 网站在后端拿到了。


# 2）隐藏式

有些应用是纯前端应用，没有后端。这时就不能用上面的方式了，必须将令牌存储在前端。


RFC 6749 就规定了第二种方式，允许直接向前端颁发令牌。这种方式没有授权码这个中间步骤，所以称为隐藏式（implicit）。

这种方式把令牌直接传给前端，是很不安全的。因此，只能用于一些安全要求不高的场景，并且令牌的有效期必须非常短，通常就是会话期间（session）有效，浏览器关掉，令牌就失效了。

流程示例：

  1. A 网站提供一个授权链接 B，要求用户跳转到 B 网站，授权用户数据给 A 网站使用。

    https://b.com/oauth/authorize?
  response_type=token&
  client_id=CLIENT_ID&
  redirect_uri=CALLBACK_URL&
  scope=read
    • response_type：值为 token，表示要求直接返回令牌。

  2. 用户跳转到 B 网站，登录后同意给予 A 网站授权。

    这时 B 网站就会跳回 redirect_uri 参数指定的跳转网址，并且把令牌作为 URL 参数，传给 A 网站。

    https://a.com/callback#token=ACCESS_TOKEN

    注意

    令牌的位置是 URL 锚点（fragment），而不是查询字符串（querystring），这是因为 OAuth 2.0 允许跳转网址是 HTTP 协议，因此存在"中间人攻击"的风险，而浏览器跳转时，锚点不会发到服务器，就减少了泄漏令牌的风险。


# 3）密码式

如果你高度信任某个应用，RFC 6749 也允许用户把用户名和密码，直接告诉该应用。该应用就使用你的密码，申请令牌，这种方式称为密码式（password）。

这种方式需要用户给出自己的用户名/密码，显然风险很大，因此只适用于其他授权方式都无法采用的情况，而且必须是用户高度信任的应用。

流程示例：

  1. A 网站要求用户提供 B 网站的用户名和密码。拿到以后，A 就直接向 B 请求令牌。

    https://oauth.b.com/token?
  grant_type=password&
  username=USERNAME&
  password=PASSWORD&
  client_id=CLIENT_ID

    • grant_type：值为 password，表示采用的授权方式是密码式。

    • username：用户名。

    • password：密码。


  2. B 网站验证身份通过后，直接给出令牌。

    注意

    这时不需要跳转，而是把令牌放在 JSON 数据里面，作为 HTTP 回应，A 因此拿到令牌。


# 4）凭证式

凭证式（client credentials），适用于没有前端的命令行应用，即在命令行下请求令牌。

这种方式给出的令牌，是针对第三方应用的，而不是针对用户的，即有可能多个用户共享同一个令牌。

流程示例：

  1. A 应用在命令行向 B 发出请求。

    https://oauth.b.com/token?
  grant_type=client_credentials&
  client_id=CLIENT_ID&
  client_secret=CLIENT_SECRET

    • grant_type：值为 client_credentials，表示采用的授权方式是凭证式。

    • client_id：客户端标识。

    • client_secret：客户端秘钥。


  2. B 网站验证通过以后，直接返回令牌。


# 5. 更新令牌

令牌的有效期到了，如果用户还在进行操作，比如正在填写一个内容很多的表单，点击保存时却因为令牌过期，跳转到了登录页，丢失了之前填写的内容，这样体验非常不好。

OAuth 2.0 允许自动更新令牌，这样在用户操作时，如果出现令牌过期的情况，自动更新一个新的令牌，让用户可以继续进行操作。

具体方法是：B 网站颁发令牌的时候，一次性颁发两个令牌，一个用于获取数据（access token），另一个用于获取新的令牌（refresh token）。令牌到期前，使用 refresh token 发一个请求去更新令牌。

https://b.com/oauth/token?
  grant_type=refresh_token&
  client_id=CLIENT_ID&
  client_secret=CLIENT_SECRET&
  refresh_token=REFRESH_TOKEN

  • grant_type：值是 refresh_token，表示要求更新令牌。

  • client_id：客户端标识。

  • client_secret：客户端秘钥。

  • refresh_token：用于更新令牌的令牌。

B 网站验证通过以后，就会颁发新的令牌。
